# PlusCal / TLA+ Guide for ostoo

Instructions for writing and checking PlusCal specs in this project.

## Toolchain

We use PlusCal (an algorithm language that compiles to TLA+) and TLC (the
model checker). Both ship with the TLA+ Toolbox.

**Install**: Download the [TLA+ Toolbox](https://lamport.azurewebsites.net/tla/toolbox.html)
to `/Applications/TLA+ Toolbox.app` (macOS). It bundles Java, `pcal.trans`,
and `tlc2.TLC` — no other dependencies needed.

**Runner script**: `specs/check.sh` wraps pcal.trans and TLC using the
Toolbox's bundled Java. See `specs/README.md` for existing specs and expected
results.

## File layout

Each spec lives in its own subdirectory under `specs/`. In-progress specs
for bug investigation live under `specs/issues/`:

```
specs/
  name/
    name.tla   — PlusCal algorithm + auto-generated TLA+ translation + properties
    name.cfg   — TLC configuration: constants, invariants, liveness properties
  issues/
    issue-name/
      name.tla — work-in-progress spec (red → green)
      name.cfg
```

| Directory | Purpose |
|---|---|
| `specs/name/` | Canonical specs — match the current Rust code, TLC passes |
| `specs/issues/issue-name/` | In-progress specs — red-green bug investigation |

| File | Purpose |
|---|---|
| `name.tla` | PlusCal algorithm + auto-generated TLA+ translation + hand-written properties |
| `name.cfg` | TLC configuration: constants, invariants, liveness properties |

## Anatomy of a `.tla` file

A `.tla` file has three sections, in order:

```
MODULE name
(* header comment *)
EXTENDS Integers, Sequences, ...
CONSTANTS ...

(*--algorithm name
  ... PlusCal code ...
end algorithm; *)

\* BEGIN TRANSLATION (auto-generated by pcal.trans — never edit this)
...
\* END TRANSLATION

\* Hand-written temporal properties go here, after END TRANSLATION
NoStarvation == ...
AllDelivered == ...

=============================================================================
```

**Key rule**: Only edit the PlusCal block (between `(*--algorithm` and
`end algorithm; *)`) and the section after `END TRANSLATION`. Never manually
edit the translation block — pcal.trans overwrites it completely.

## Workflow: Red-Green Specification

Specs follow a red-green workflow similar to test-driven development. You
first model the suspected bug ("red"), confirm TLC catches it, then iterate
on the spec until it passes ("green"), implement the fix in Rust to match,
and finally promote the passing spec to the base spec location.

### Overview

```
1. Model the bug        → specs/issues/<issue>/   (red — TLC finds violation)
2. Fix the spec         → specs/issues/<issue>/   (green — TLC passes)
3. Implement in Rust    → match the green spec
4. Promote the spec     → copy to specs/<name>/   (replaces base spec)
5. Tag correspondence   → update [spec:] tags in Rust code
```

### Step 1: Model the bug (red)

Create a new subdirectory under `specs/issues/` named after the issue or
bug being investigated:

```bash
mkdir -p specs/issues/lost-wakeup
```

Write a PlusCal spec that faithfully models the **current (buggy)** code.
Include the safety/liveness properties you expect to be violated:

```bash
specs/issues/lost-wakeup/
  completion_port.tla    # models the buggy protocol
  completion_port.cfg    # invariants + liveness properties
```

Run TLC and confirm it finds the violation:

```bash
./check.sh issues/lost-wakeup/completion_port
```

TLC should report a counterexample trace. This is the "red" phase — the
spec proves the bug exists. If TLC passes, your model doesn't capture the
race; revisit the label boundaries (atomicity).

### Step 2: Fix the spec (green)

Edit the PlusCal code in the same `specs/issues/<issue>/` directory to model
the proposed fix. For example, move `mark_blocked` into the same label as
`set_waiter` to model doing both under a single lock:

```
\* Before (buggy — two labels):
SetWaiter:
    waiter := self;
MarkBlocked:
    thread_state := "blocked";

\* After (fixed — one label):
CheckAndAct:
    waiter := self;
    thread_state := "blocked";
```

Re-run TLC until all properties pass:

```bash
./check.sh issues/lost-wakeup/completion_port
```

This is the "green" phase — the spec proves the fix is correct.

### Step 3: Implement in Rust

Write the Rust code to match the green spec. The PlusCal labels map to lock
scopes — operations in the same label must be under the same lock in Rust.
See [Mapping PlusCal to Rust code](#mapping-pluscal-to-rust-code) below.

### Step 4: Promote the spec

Once the Rust implementation matches and the spec passes, copy the green
spec from `specs/issues/` to the base spec location:

```bash
cp specs/issues/lost-wakeup/completion_port.tla specs/completion_port/completion_port.tla
cp specs/issues/lost-wakeup/completion_port.cfg specs/completion_port/completion_port.cfg
```

Update the MODULE name inside the `.tla` file if it changed. Verify the
base spec still passes:

```bash
./check.sh completion_port
```

The issue directory can be deleted or kept for historical reference.

### Step 5: Tag correspondence

Update `// [spec: name/name.tla Label]` comments in the Rust code to match
the promoted spec. Use `grep -rn '\[spec:' .` to find all tags.

### Quick reference (editing a spec without a bug)

For routine spec edits that don't start from a bug:

1. Edit the PlusCal code between `(*--algorithm` and `end algorithm; *)`.
2. Run `./check.sh name` — this translates PlusCal and runs TLC.
3. If TLC finds a violation, read the trace, fix, re-run. Repeat.

pcal.trans rewrites the `BEGIN TRANSLATION` ... `END TRANSLATION` block
in-place. It uses checksums to detect changes. You never need to manually
delete the translation block.

## Writing PlusCal specs

### Modelling atomicity

Each PlusCal **label** is one atomic step. Code between two labels executes
without interleaving from other processes. Use this to model lock scopes:

```
\* Everything in one label = one lock acquisition
CheckAndAct:
    if condition then
        \* drain
    else
        waiter := self;
        thread_state := "blocked";
    end if;

\* Separate label = after releasing the lock — other processes can interleave
WaitUnblocked:
    await thread_state = "running";
```

If two operations happen under the same lock in the real code, they **must**
be in the same label. If they happen outside a lock (or across an
unlock/relock boundary), they **must** be in separate labels. Getting this
wrong will either hide real bugs or produce false positives.

### Process types

```
\* Multiple instances (one per element of set)
fair process Producer \in 1..NUM_PRODUCERS

\* Single instance
fair process Consumer = 0
```

Use `fair process` for processes that should not halt forever (enables weak
fairness for liveness checking). All process IDs must be the same type
(all integers or all strings) — TLC cannot mix types in ProcSet.

### Variables

- **Global variables**: shared state (queues, counters, flags)
- **Process-local variables**: declared inside `process`, automatically
  indexed by `self`
- **Ghost variables**: exist only in the spec for verification (e.g.
  `consumed_values` set for checking no-duplicate-delivery)

### Common patterns

**Blocking wait** (models `block_current_thread` + `unblock`):

```
thread_state := "blocked";
...
await thread_state = "running";  \* blocks until a producer sets this
```

**Non-deterministic choice** (models "any of these could happen"):

```
with s \in {TRUE, FALSE} do
    p_simple := s;
end with;
```

**Bounded queue with backpressure**:

```
if Len(queue) < MAX_QUEUED then
    queue := Append(queue, item);
else
    total_dropped := total_dropped + 1;
end if;
```

### Safety invariants

Defined in a `define` block inside the algorithm. Checked in every reachable
state.

```
define
    QueueBounded == Len(queue) <= CAPACITY
    HeadNeverPassesTail == head <= tail
    AccountingSound ==
        total_posted = total_delivered + Len(queue) + total_dropped
end define;
```

### Liveness properties

Defined **after** `END TRANSLATION` (not inside the algorithm). Referenced in
the `.cfg` file under `PROPERTY`.

```
\* "Every blocked thread is eventually unblocked"
NoStarvation ==
    [](thread_state = "blocked" => <>(thread_state = "running"))

\* "All items are eventually delivered"
AllDelivered ==
    <>(total_delivered + total_dropped = TOTAL_EXPECTED)
```

`[]` means "always", `<>` means "eventually", `=>` means "implies",
`~>` means "leads to".

## `.cfg` file format

```
\* TLC configuration
SPECIFICATION Spec

CONSTANTS
    NUM_PRODUCERS = 2
    MAX_QUEUED = 4
    MIN_COMPLETE = 1
    MAX_PRODUCE = 2

INVARIANT
    SafetyInvariant

PROPERTY
    NoStarvation
    AllDelivered
```

Keep constants small for fast iteration. State space grows exponentially —
doubling `MAX_PRODUCE` can 10x+ the states explored. Start small, increase
once the spec passes to gain more confidence.

## Mapping PlusCal to Rust code

When writing a spec, document the correspondence between PlusCal constructs
and Rust code. This makes it possible to verify that the spec actually models
what the code does.

Key things to map:
- **Labels ↔ lock scopes**: which operations are atomic in the real code?
- **Variables ↔ fields**: which struct fields / atomics does each variable represent?
- **Process types ↔ threads/contexts**: which code paths does each process model?
- **await ↔ blocking primitive**: what does the real blocking mechanism look like?

See `README.md` for the existing correspondence tables.

### Cross-referencing from Rust code

Tag Rust code with `// [spec: file.tla Label]` comments to link back to the
PlusCal spec. This makes the mapping greppable from both sides.

```rust
// [spec: spsc_ring/spsc_ring.tla AcquireHead]
let head = hdr.head.load(Ordering::Acquire);
let tail = hdr.tail.load(Ordering::Relaxed);

// [spec: spsc_ring/spsc_ring.tla CheckFull]
if tail.wrapping_sub(head) >= self.cq_entries {
    return false;
}
```

Use `grep -rn '\[spec:' .` to find all annotated sites. When updating a spec
label name, update the Rust tags to match.

## Tips

- **Start with safety**: get invariants passing before adding liveness
  properties. Liveness checking is slower and harder to debug.
- **Use ghost variables** for properties that need to track history (e.g. "no
  duplicate delivery" needs a set of consumed values).
- **Model the bug first** (red-green workflow): if you suspect a race,
  create a spec under `specs/issues/` that models the current (potentially
  buggy) code. If TLC finds the bug (red), fix the spec until it passes
  (green), implement the fix in Rust, then promote the spec to `specs/`.
  See [Workflow: Red-Green Specification](#workflow-red-green-specification).
- **Don't over-model**: abstract away details that don't affect the property
  you're checking. The spec doesn't need to be a 1:1 translation of the
  Rust code — it needs to faithfully model the concurrency protocol.
- **Beware the `Termination` name**: PlusCal auto-generates a `Termination`
  property. Don't define your own with that name — use something like
  `AllDelivered` instead.
